# Project Bootstrapping

## 适用时机

- 架构已定，需要建立最小可持续工程骨架。

## 输入

- 项目类型与包边界。
- 当前仓库已有工具链或空仓库状态。

## 动作

1. 先定目录结构、入口文件、脚本约定、环境变量边界。
2. 统一 `tsconfig` 分层，至少保证基础配置、应用配置和测试配置职责清晰。
3. 对 `web-app` 默认采用 Vite + Rolldown，准备 `dev`、`build`、`preview`、`lint`、`typecheck`、`test` 脚本。
4. 对 `ts-library` 准备库入口、子路径导出、类型声明输出、测试入口和变更约束。
5. 对 `node-app` 准备运行入口、构建入口、开发模式和生产模式脚本。
6. 把 lint、typecheck、test 接进日常脚本，不要把质量门槛留到“最后有空再补”。
7. 如果仓库已有稳定脚手架或 CI 约束，复用它；只在它妨碍目标时才提出替换。

## 输出

- 一套能启动、能检查、能测试、能构建的工程骨架。
- 一套统一脚本名称和目录规则。
- 一份环境变量和配置来源说明。

## 完成定义

- 新人进入项目时，不需要猜测命令和入口。
- 工程最小闭环已经成立：能启动、能检查、能测试、能构建。
- 基础结构已经能承接 UI、业务逻辑和共享包的沉淀。

## 反模式

- 目录结构先随手写，后面再“慢慢整理”。
- 把运行配置散落在代码常量、脚本参数和环境变量里。
- 开发脚本和构建脚本各写一套逻辑，彼此行为不一致。
