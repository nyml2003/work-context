# Maintenance And Evolution

## 适用时机

- 项目已经上线或已经被团队持续使用，需要扩展、治理或升级。

## 输入

- 现有代码、组件复用情况、依赖版本、性能反馈、设计系统债务。

## 动作

1. 先判断问题属于功能债、样式债、组件债、构建债，还是依赖债。
2. 新功能优先沉淀到共享层；确认没有共享价值时再做局部实现。
3. 定期回收重复样式、重复组件和重复工具函数，避免 UI 库名义存在但实际失效。
4. 升级 React、TypeScript、构建工具或测试工具时，先做兼容面盘点，再做分阶段迁移。
5. 对性能和稳定性问题，优先收敛结构复杂度与无效渲染，再谈局部微优化。
6. 把例外方案登记为显式决策，不让临时补丁演变成默认模式。

## 输出

- 一份治理清单，说明要保留、要淘汰、要迁移、要沉淀的内容。
- 一份增量演进计划，说明分批次推进方式和回退点。
- 一份关键架构决策记录。

## 完成定义

- 团队知道接下来要清什么债，为什么清，风险在哪里。
- 设计系统和共享逻辑持续增长，而不是被业务页面绕开。
- 版本升级和结构调整有节奏，不会每次都变成大爆炸重写。

## 反模式

- 发现重复问题却一直靠复制粘贴和局部 patch 维持。
- 把组件库和样式体系当作一次性工程，而不是持续维护的基础设施。
- 遇到演进问题只会整体重写，不会做分阶段迁移。
