# Development Workflow

## 适用时机

- 已有骨架和设计系统，需要推进具体功能、页面或共享逻辑。

## 输入

- 当前功能需求、接口契约、现有组件和公共逻辑。

## 动作

1. 先拆交互流、数据流和状态流，再落组件。
2. 让容器、展示、控制逻辑保持分工清晰，避免组件同时承担请求、状态、布局和视觉细节。
3. 对 `web-app`：
   - 覆盖加载态、空态、错误态、权限态、响应式断点和键盘可达性。
   - 优先复用共享 UI 和共享逻辑，不在页面里重复发明表单、列表、弹层模式。
4. 对 `ts-library`：
   - 先锁 API 形状、类型推导、错误语义和可测试性。
   - 保持纯逻辑、可组合、低副作用。
5. 对 `node-app`：
   - 把输入解析、业务处理、输出渲染和运行时适配分开。
   - 明确日志、错误、超时、退出码或服务生命周期处理。
6. 需要跨层改动时，先调整共享层，再回填业务层，不要把一次性 patch 留成长期结构。

## 输出

- 一组按功能切片落地的实现。
- 一份说明当前 feature 依赖了哪些共享层，新增了哪些共享能力。
- 一份可联调、可调试、可回归验证的结果。

## 完成定义

- 关键路径已经可运行且结构清晰。
- 共享逻辑沉淀到正确层级，没有把公共代码埋进页面或命令入口。
- 每次功能改动都有明确的回归面。

## 反模式

- 先把功能写出来，再回头补边界和状态设计。
- 组件过大，既处理请求又处理样式还处理业务决策。
- 发现共享需求后仍然复制粘贴，而不是收敛到公共层。
