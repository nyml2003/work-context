# Plan Issue Taxonomy

Use this reference to decide whether a conversation contains a reusable planning issue worth summarizing and storing.

## Strong candidates

### Missing goal or success criteria

- Signals:
  - The user asks for a plan before the desired outcome is explicit.
  - The same "what does success look like?" question keeps returning.
- Reusable response:
  - Lock goal, success criteria, and out-of-scope items before detailed planning.

### Scope ambiguity or repeated clarification

- Signals:
  - The plan keeps changing because core scope boundaries were never fixed.
  - The same scope question is asked more than once.
- Reusable response:
  - Confirm scope, non-goals, and constraints in a compact checklist before decomposing work.

### Hidden assumptions

- Signals:
  - Implementation or planning choices keep getting revised because defaults were implied, not stated.
  - The conversation reveals incompatible assumptions late.
- Reusable response:
  - State the chosen defaults up front and flag anything still uncertain before execution.

### Weak handoff boundaries

- Signals:
  - Plans fail because roles, ownership, or acceptance criteria were too vague.
  - Review or downstream execution reopens already-settled questions.
- Reusable response:
  - Emit concrete handoff artifacts: owner, scope, acceptance criteria, and open questions.

### Missing review or closure loop

- Signals:
  - The task ends without a summary of what was learned.
  - The same process mistake comes back across conversations.
- Reusable response:
  - Add a short retrospective step that captures what should change next time.

## Do not store

- One-off project facts
- Temporary personal preferences
- Raw transcripts or long conversation logs
- Rules that are useful only for the current task
