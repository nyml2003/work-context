# 模式选择

## 单 agent 路线

优先使用单 agent 的情况：

- 只涉及单一主工作流，例如纯 UI 调整、单个页面实现、局部逻辑修补。
- 主要价值在快速落地，而不是在多人并行。
- 改动面集中，文件 ownership 清晰，不值得拆成多个 handoff。
- 用户没有要求并行，也没有明显的多角色协作价值。

## 多 agent 路线

优先使用多 agent 的情况：

- 同时涉及 UI、逻辑、API、日志等多个独立工作面。
- 存在明确的设计评审、约束总结、实现、测试、回归评审阶段。
- 任务规模足够大，拆分后能减少上下文污染或并行推进。
- 用户明确要求子 agent 或并行推进。

## 默认判断

1. 如果不确定，先偏向单 agent。
2. 如果主工作流超过一个，并且下游 skill 的装配已经明显不同，再切到多 agent。
3. 如果拆分后每个子任务仍然高度耦合，说明还不该并行。

## 输出

- 当前选择单 agent 还是多 agent
- 选择原因
- 下一步要进入的内部模块或子任务阶段
