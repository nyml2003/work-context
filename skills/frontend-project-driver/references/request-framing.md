# 请求 framing

## 适用时机

- 用户直接说“做一个页面”“做一个前端项目”“重构这个模块”，但没有给出明确任务结构。
- 需要先判断现在是在做新建、迭代、重构、联调还是验收。
- 需要把自然语言需求整理成后续 skill 可以消费的最小任务对象。

## 先收集什么

1. 目标是什么，用户要看到什么结果。
2. 当前是在新建项目、已有仓库迭代、局部重构，还是收尾验收。
3. 主要工作流涉及哪些面：UI、逻辑、API、日志、设计评审、测试评审。
4. 限制条件有哪些：TypeScript、TSX、Tailwind、ESLint、现有设计系统、现有 API 契约。
5. 成功标准和验收标准是什么。

## 最小任务结构

如果后续需要进入 agent 编排或 CLI，可先整理成下面这些字段：

- `title`
- `objective`
- `workstreams`
- `constraints`
- `acceptance_criteria`
- `notes`

## 默认规则

- 不要一开始就写 `task.json`，先把需求说清楚。
- 能从现有仓库和用户请求中推断出来的内容，先推断，不要把整理工作甩给用户。
- framing 的目标是给后续路由服务，不是做长篇需求文档。
