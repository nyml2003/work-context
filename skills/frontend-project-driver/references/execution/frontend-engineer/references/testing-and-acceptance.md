# Testing And Acceptance

## 适用时机

- 功能基本完成，需要确认质量门槛和交付条件。

## 输入

- 当前实现、风险点、关键用户路径、构建产物。

## 动作

1. 默认先跑 `lint`、`typecheck`、`test`。
2. 对 `web-app`，再补关键流程交互测试和基础 a11y 检查。
3. 对 `ts-library`，重点覆盖核心算法、类型边界、错误路径和导出面。
4. 对 `node-app`，重点覆盖命令入口、配置解析、错误处理和主要运行路径。
5. 验收时同时检查正常路径、错误路径、边界条件和回归风险。
6. 如果某项测试暂时无法落地，明确记录缺口、原因和补齐条件，不要把缺口伪装成通过。

## 输出

- 一份明确的测试结果。
- 一份明确的验收结论：通过、带风险通过，或未通过。
- 一份剩余风险清单。

## 完成定义

- 质量门槛已经能证明当前交付不是“只在开发者机器上能跑”。
- 已经知道哪些风险被覆盖，哪些风险仍然暴露。
- 团队能基于结果做发布或继续迭代决策。

## 反模式

- 构建通过就视为验收完成。
- 只有 happy path，没有错误路径和边界条件。
- 明知存在测试缺口，却不留下后续动作。
