# Build And Release

## 适用时机

- 需要把开发结果转成可交付产物。

## 输入

- 项目类型、运行环境、部署方式、消费者类型、版本策略。

## 动作

1. 默认沿用 Vite + Rolldown 路线规划构建体验和产物边界。
2. 对 `web-app`：
   - 输出浏览器可部署产物。
   - 关注代码分包、静态资源、source map、部署环境变量、预览和回滚边界。
3. 对 `ts-library`：
   - 输出清晰的导出面和类型声明。
   - 优先保证 ESM 体验；只有在消费者明确需要时再补 CJS 兼容。
   - 明确 `exports`、副作用边界和版本变更影响。
4. 对 `node-app`：
   - 明确运行入口、可执行入口、外部依赖、Node 内建模块处理和部署方式。
   - 让开发模式、构建模式和发布模式的行为一致。
5. 把发布前检查、版本信息、变更说明和回滚条件纳入交付流程。

## 输出

- 一份按项目类型定义清楚的构建产物。
- 一份发布说明，包含产物、版本、兼容边界和已知限制。
- 一份回滚或回退条件说明。

## 完成定义

- 消费者已经知道如何安装、部署或运行产物。
- 构建产物边界清楚，没有混入测试专用代码或临时调试配置。
- 发布流程可重复，不依赖操作者临场记忆。

## 反模式

- 开发环境能跑就算“可发布”。
- 库产物没有明确导出面，消费者只能靠猜。
- Node 项目没有区分源码入口、运行入口和打包入口。
