# Architecture

## 适用时机

- 范围已经收敛，需要确定系统边界、模块划分和渲染方式。

## 输入

- `discovery-and-scoping.md` 产出的范围说明。
- 当前仓库已有目录、包结构、依赖和运行方式。

## 动作

1. 默认使用 React LTS 作为协调层，负责状态编排、交互组织和视图协调。
2. 把渲染边界单独判断：DOM、Canvas、Three、CLI 或其他 renderer 按场景再选，不要把 React 和具体渲染细节绑死。
3. 把代码至少拆成应用层、领域逻辑层、基础设施层、UI 层。
4. 对 `web-app`：
   - 先定路由、页面壳层、数据获取、状态归属和缓存边界。
   - 把页面容器、展示组件、共享组件分开。
5. 对 `ts-library`：
   - 保持纯逻辑边界，不向外泄漏 DOM 或 Node 专属依赖。
   - 先定导出面、类型面、错误语义和副作用边界。
6. 对 `node-app`：
   - 把命令入口、服务入口、任务入口和适配层分开。
   - 把运行配置、日志、错误处理、进程生命周期单独收口。
7. 需要多包时，优先按共享价值拆包：`app`、`ui`、`shared`、`tooling`，不要为了“像 monorepo”而强行拆细。

## 输出

- 一份模块图或文字版架构说明。
- 一份包边界说明，明确哪些能力共享、哪些能力只在当前项目内部使用。
- 一份渲染边界说明，明确 React 与 renderer 的职责线。

## 完成定义

- 已经能解释每个目录或包为什么存在。
- 已经能定位共享逻辑、共享 UI、业务逻辑和运行时适配代码。
- 后续工程初始化不需要再临时发明目录和边界。

## 反模式

- 把页面、状态、网络请求、工具函数和样式揉在同一层。
- 为了“先进”而提前拆过多包，导致维护成本高于复用收益。
- 没有定义导出面和依赖边界，导致库代码与应用代码相互污染。
