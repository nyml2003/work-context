# Styling And UI

## 适用时机

- 需要建立或扩展设计系统、样式体系、组件库。

## 输入

- 品牌基调、交互风格、已有视觉资产、现有组件和样式债务。

## 动作

1. 先定义 design tokens：颜色、字号、字重、行高、间距、圆角、阴影、层级、动效、断点。
2. 再把 tokens 落成 CSS 原子类，要求命名稳定、可组合、可搜索、可复用。
3. 再建设 UI 分层：
   - `tokens`
   - `atomic classes`
   - `primitives`
   - `composites`
   - `feature components`
4. 优先手动维护样式和 UI 库，不默认引入大型 UI 框架或黑盒样式体系。
5. 页面开发时优先复用原子类和基础组件；只有确定共享价值不足时才写局部样式。
6. 设计复杂交互时，先把状态和结构抽清楚，再决定动画和视觉细节。

## 输出

- 一份 token 清单。
- 一套可以稳定复用的 CSS 原子类基础层。
- 一套从基础组件到组合组件的自维护 UI 库。

## 完成定义

- 新页面能优先通过已有 tokens 和原子类搭出骨架。
- 基础组件具备一致的尺寸、间距、交互和禁用态约束。
- 样式修改不会迫使业务页面逐个手改。

## 反模式

- 直接在页面里堆匿名样式，事后再抽。
- 把组件库做成页面片段仓库，而不是稳定的 UI 基础设施。
- 在没有 design tokens 的前提下先铺大量样式类。
