# Discovery And Scoping

## 适用时机

- 新项目从 0 到 1。
- 现有项目需求模糊、边界不清或需要重新定义成功标准。
- 接手遗留项目但还不知道该按 `web-app`、`ts-library` 还是 `node-app` 推进。

## 输入

- 用户目标、现有仓库状态、运行环境约束、交付时间。
- 是否已有设计系统、构建体系、测试体系、部署体系。

## 动作

1. 先判断工作模式：`new-project`、`iteration`、`governance`。
2. 再判断项目类型：`web-app`、`ts-library`、`node-app`。
3. 明确主要用户、核心任务、关键页面或关键 API、运行时、兼容边界、性能目标、可访问性要求。
4. 明确 React 只承担协调职责，渲染层是否落在 DOM、Canvas、Three、CLI 或其他场景可以后定。
5. 列出明确的不做项，避免把需求无限扩张到“顺手一起做”。

## 输出

- 一份可执行的范围说明，至少包含目标、非目标、受众、运行环境、成功标准、主要风险。
- 一份阶段判断，说明当前优先进入哪个生命周期步骤。
- 一份项目类型判断，说明后续按 Web 应用、纯逻辑 TS 工具库，还是 Node 项目规范推进。

## 完成定义

- 已经能在不猜测的前提下进入架构设计。
- 已经知道哪些内容属于本次交付，哪些内容明确不做。
- 已经知道交付的主产物是浏览器界面、纯逻辑包，还是 Node 运行产物。

## 反模式

- 需求还没收敛就直接开始堆页面或堆配置。
- 不区分生命周期阶段，把探索、设计、实现、验收混成一锅。
- 对已有工程背景视而不见，默认按全新项目处理。
