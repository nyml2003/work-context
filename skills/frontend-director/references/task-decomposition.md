# 任务拆解

## 适用时机

- 一个前端任务已经大到不能由单个 worker 稳定完成。
- 用户明确提出并行、多 agent 或分角色推进。
- 任务同时涉及 UI、逻辑、接口、日志、评审中的多项。

## 拆解原则

1. 先按结果拆，不按代码层名词拆。
2. 先按责任拆，不按目录拆。
3. 每个子任务都要有清晰的输入和输出。
4. 不要把“看一看”“顺手查一下”拆成独立子任务。
5. 一个子任务默认只绑定一个主 capability。
6. 如果两个子任务的主要价值都依赖同一份决策，先把那份决策提升为上游 policy 或 director 输出。

## 推荐子任务形态

- `director`: 负责总拆解、依赖和并行规划。
- `policy`: 负责约束总结、设计判断和边界定义。
- `worker`: 负责某一类实现面。
- `review`: 负责测试覆盖、回归风险和发布准备。

## 反模式

- 一个 worker 同时背负多个主 capability。
- 为了凑 agent 数量而制造子任务。
- 子任务之间没有 handoff，只能靠重复阅读完整上下文继续工作。
